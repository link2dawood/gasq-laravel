## 1. Project description

**GASQ Laravel** is a web application for running a security-services marketplace: organizations can post work, vendors can participate, and both sides can use built-in pricing and operations calculators. The product combines marketing pages, user accounts, job listings, optional credit-based access to tools, and administrative featuresÔÇöimplemented as a standard Laravel site with server-rendered pages and a conventional database-backed architecture.

---

## 2. Platform operation ÔÇö how the system works

The subsections below cover user journeys, security gates, calculators, billing, and supporting services.

### 2.1 What this application is

GASQ Laravel is a **web application** for the GASQ security-services marketplace. It provides:

- **Marketing and informational pages** (landing, pricing, FAQ, pay scale, payment model).
- **User accounts** (buyers, vendors, admins) with profile and wallet **credits**.
- **Job board** (post jobs, bids, vendor profiles).
- **Security and labor calculators** implemented as **Blade + Bootstrap** pages with **server-side math** (PHP ÔÇťV24ÔÇŁ compute engines), not a React shell for the main calculator experience.
- **Monetization** via **Stripe** checkout to add credits to a user wallet, and **per-run deduction** of credits when a calculator **compute API** succeeds.
- **SMS OTP** for **phone verification** after registration (Twilio, custom codes stored in the database).

Core persistence uses **Laravel + Eloquent** (MySQL or other configured database). Session-based authentication is standard Laravel `Auth`.

### 2.3 Roles

- **Buyer**: Purchases security services; can post jobs and use calculators per business rules.
- **Vendor**: Can bid and use calculators; **not** required to post a job to unlock calculators.
- **Admin**: Analytics, settings, FAQ, token/wallet management (routes under `admin` + `EnsureUserIsAdmin`).

User model helpers (e.g. `isBuyer()`, `isVendor()`, `isAdmin()`) support middleware and policies.

### 2.4 End-to-end user flow (buyer-centric)

This is the intended **happy path** for someone who wants to use paid calculators:

1. **Register / log in**  
   Standard Laravel registration (buyer vs vendor per registration path / `user_type`).

2. **Verify phone (SMS OTP)**  
   After signup, the user completes **`/phone/verify`**: codes are generated by the app, stored (e.g. hashed) in **`verification_codes`**, and sent via **Twilio**. Middleware **`phone.verified`** blocks downstream features until `users.phone_verified` is set.

3. **Purchase credits**  
   User opens **`/credits`**, chooses a **pricing plan**, and pays with **Stripe Checkout**. Webhook **`/stripe/webhook`** confirms payment and **`WalletService`** credits the wallet; transactions are recorded for history and receipts.

4. **Post at least one job (buyers only)**  
   Middleware **`buyer.has_job`** requires that a **`job_postings`** row exists for that user before calculator **page** routes in the gated group are allowed. Vendors skip this requirement.

5. **Use calculators**  
   User opens a calculator URL (see ┬ž2.6). The page loads inputs; changing inputs triggers **POST** requests to **`/_backend/.../compute`** endpoints. On **success**, **`CalculatorRunBillingService`** deducts **`config('credits.calculator_per_run')`** credits (from **`.env`**: **`CALCULATOR_CREDITS_PER_RUN`**, default 5, minimum 1) inside a **database transaction** so a failed engine run does not consume credits.

6. **Review balance**  
   **`/account-balance`** shows transaction history; **`/credits/***` supports buying more.

**Credits page copy** explains that each calculator run uses the configured number of credits (see `resources/views/credits/index.blade.php`).

### 2.5 Middleware and access control

Aliases registered in **`bootstrap/app.php`**:

| Alias | Purpose |
|-------|--------|
| `phone.verified` | Redirects to phone verification if not verified. |
| `has.credits` | Requires wallet balance **Ôëą** `config('credits.calculator_per_run')` so users cannot open calculator UIs without enough credits for one run. |
| `buyer.has_job` | For **buyers** only, requires at least one `JobPosting`; vendors pass through. |

**Important routing split** (see **`routes/web.php`**):

- **Calculator Blade pages** (main menu, contract analysis, cost analysis, bill rate, instant estimator index, etc. **inside** the group at line ~81) use:  
  `auth` + `phone.verified` + `has.credits` + `buyer.has_job`.
- **`/_backend/.../compute`** POST routes live under **`auth` + `phone.verified`** only (same session as the app). Credits are still enforced at **deduction** time; buyers who bypass the UI would not be blocked by `has.credits` or `buyer.has_job` on the HTTP route aloneÔÇö**tightening** that (e.g. attaching the same middleware to compute routes) is a recommended hardening step if direct API access is a concern.

**Public calculator-related pages** (no login required for **view**): examples include **`/security-quote`** (marketing preview; no server quote math), **`/calculator`** hub, **`/gasq-tco-calculator`** **view**, **`/post-job`** informational, etc. **Compute** for those tools still requires an authenticated, verified user when using the backend API.

### 2.6 Calculator surface area

### 2.6.1 Gated full calculators (typical ÔÇťappÔÇŁ calculators)

These **GET** routes expect the full middleware stack on the **page** (auth, phone, credits, buyer job):

- Main menu, contract analysis, security billing, mobile patrol calculator & comparison  
- Cost analysis, bill rate analysis, manpower hours, economic justification  
- Hourly pay, budget, unarmed guard services  
- Mobile patrol **analysis** & **global security pricing**  
- Mobile patrol hit calculator, buyer fit index  
- Workforce appraisal report (and aliases CFO / post position / appraisal comparison)  
- GASQ direct labor build-up, GASQ additional cost stack  

Navigation is centralized in partials such as **`resources/views/partials/nav-calculators-dropdown.blade.php`**.

### 2.6.2 Other entry points

- **`/gasq-instant-estimator`**: Combined **public** GET with **`/instant-estimator`** redirect; **compute** uses **`/_backend/instant-estimator/compute`** (authenticated).
- **Static/marketing** pages: **`/post-coverage-schedule`**, **`/open-bid-offer`**, **`/vendor-form`**, **`/register/buyer`**, **`/register/vendor`**, etc., are largely Blade content (some link into authenticated job creation).

### 2.6.3 Backend compute endpoints

Authenticated **`POST`** JSON payloads (`version: v24` where applicable, `scenario` array) hit controllers under **`App\Http\Controllers\Backend\`**, for example:

- **`StandaloneV24ComputeController`** ÔÇö `type` in route must match **`StandaloneV24ComputeService`** (cost analysis, bill rate, MPA, GSP, workforce appraisal, TCO, absorbed rate, government, keeps doors open, unarmed, buyer fit, direct labor build-up, additional cost stack, etc.).
- **`InstantEstimatorComputeController`**, **`MainMenuComputeController`**, **`ContractAnalysisV24ComputeController`**, **`MobilePatrolV24ComputeController`**, **`SecurityBillingV24ComputeController`**, **`SecurityBillingComputeController`**.

Successful responses include calculator payload plus **`credits_spent`** and **`credits_remaining`** (when billing applies). **HTTP 402** JSON is returned when the wallet cannot cover the run.

### 2.7 Credits and configuration

- **Wallet**: `wallets.balance` per user; **`WalletService`** for **add** (purchase, bonus, admin) and **spend**.
- **Transactions**: Ledger rows for audit (`transactions` table).
- **Per-run cost**: **`config/credits.php`** ÔćÉ **`CALCULATOR_CREDITS_PER_RUN`** in **`.env`** (documented in **`.env.example`**).
- **Stripe**: See the Stripe environment section (┬ž6) of this document and **`StripeCreditsController`**; success flow may nudge buyers toward **`jobs.create`** if they have credits but no job yet (see **`CreditsController`**).

Admin can inspect/adjust wallets via **`/admin/tokens`** ( **`AdminTokensController`** ).

### 2.8 Phone verification (OTP)

- Registration collects **phone**; after account exists, user completes OTP at **`/phone/verify`**.
- **Channel**: SMS only via Twilio; **provider mode**: **custom** codes generated in-app and persisted (**not** Twilio Verify API for the code itself).
- Env vars: **`TWILIO_ACCOUNT_SID`**, **`TWILIO_AUTH_TOKEN`**, **`TWILIO_FROM`** (see **`.env.example`**).

### 2.9 Testing and quality

- **PHPUnit**: Unit tests for standalone engines and **`tests/Unit/StandaloneV24ParityTest.php`** (golden JSON fixtures under **`tests/Fixtures/v24/standalone/`**).
- **Docs**: Section 10 (Spreadsheet Ôćö UI cell mapping) relates workbook/React parity notes to engines where applicable.

### 2.10 Summary for stakeholders

**GASQ Laravel** delivers a **buyerÔÇôvendor marketplace** with **credit-based access** to **professional security calculators**. Buyers **verify their phone**, **buy credits with Stripe**, **post a job**, then use **native Blade calculators** backed by **server-side PHP engines**. Each **successful** calculator **API run** costs **`CALCULATOR_CREDITS_PER_RUN`** credits (configurable). Vendors can use calculators without posting a job. The system is **documented, testable**, and **operable** without relying on a React SPA for core calculator UX.

## 3. Setup

## Prerequisites

- PHP 8.2+
- Composer
- Node.js & npm (for front-end assets)
- SQLite (default) or MySQL/PostgreSQL

## Local setup (native)

1. Clone the repository and enter the Laravel project:

   ```bash
   cd gasq-laravel
   ```

2. Install PHP dependencies:

   ```bash
   composer install
   ```

3. Copy environment file and generate key:

   ```bash
   cp .env.example .env
   php artisan key:generate
   ```

4. Configure `.env` (database, mail, optional Google OAuth). Default uses SQLite; ensure `database/database.sqlite` exists or run:

   ```bash
   touch database/database.sqlite
   ```

5. Run migrations:

   ```bash
   php artisan migrate
   ```

6. (Optional) Build front-end assets:

   ```bash
   npm install && npm run build
   ```

7. Start the dev server:

   ```bash
   php artisan serve
   ```

   Open `http://localhost:8000`.

## Docker (local only)

Docker is for **local development only**, not production.

### Quick start

1. From the project root (`gasq-laravel`):

   ```bash
   cp .env.docker.example .env
   php artisan key:generate
   ```

   Or copy your existing `.env` and set `DB_HOST=db`, `DB_DATABASE=gasq`, `DB_USERNAME=gasq`, `DB_PASSWORD=secret`, `APP_URL=http://localhost:8080`.

2. Build and start containers:

   ```bash
   docker compose up -d --build
   ```

3. On first run, the app container will run `composer install` automatically if `vendor` is missing. Generate key and run migrations:

   ```bash
   docker compose exec app php artisan key:generate
   docker compose exec app php artisan migrate
   ```

4. (Optional) Build assets:

   ```bash
   docker compose exec node npm install && docker compose exec node npm run build
   ```

   Or run `npm run build` on the host and ensure `public/build` is present.

5. Open the app at **http://localhost:8080**.

### Services

| Service | Purpose |
|---------|---------|
| `app` | PHP 8.2 + Laravel; serves the app (artisan serve or PHP-FPM) |
| `db` | MySQL 8 (or set DB_CONNECTION=sqlite and use a volume) |
| `node` | Optional; for running npm build inside the stack |

## 4. Development

## Workflow

1. Create a feature branch from `main` (or your default branch).
2. Implement following the plan phases: auth extension Ôćĺ data model Ôćĺ landing/content Ôćĺ marketplace Ôćĺ calculators Ôćĺ admin Ôćĺ reports.
3. Run tests and lint before pushing.
4. Document new modules or env vars in `docs/` and update `docs/modules.md` status.

## Commands

```bash
# Run migrations
php artisan migrate

# Rollback last migration
php artisan migrate:rollback

# Run seeders
php artisan db:seed

# Clear caches
php artisan config:clear && php artisan cache:clear && php artisan view:clear

# Run queue worker (if using queue)
php artisan queue:work

# Run tests
php artisan test
# or
./vendor/bin/phpunit
```

## Code style

- Use Laravel Pint for PHP formatting: `./vendor/bin/pint`.
- Controllers: keep thin; put business logic in `app/Services`.
- Use Form Requests for validation.
- Use Eloquent relationships and scopes; avoid raw SQL unless needed.

## Blade and Bootstrap

- Shared UI: use Blade components in `resources/views/components/`.
- Prefer Bootstrap 5 utility classes and components for layout and forms.
- Match existing GASQ UI/UX from the reference design where specified in the plan.

## Adding a new module

1. Add migrations for any new tables.
2. Create Eloquent models and relationships.
3. Add service class(es) in `app/Services` if needed.
4. Add controller(s) and routes.
5. Add Blade views; use shared layout and components.
6. Update `docs/modules.md` and any relevant docs.

## 6. Stripe environment

The app uses **Stripe mode** (test vs live) to choose which API keys and webhook secret to use.

## How it works

1. **`APP_ENV`** ÔÇô LaravelÔÇÖs environment: `local`, `staging`, `production`, etc.
2. **`STRIPE_MODE`** (optional) ÔÇô Override for Stripe only: `test` or `live`.
   - If **set**: that value is used.
   - If **not set**: `production` Ôćĺ `live`, anything else Ôćĺ `test`.

So you can either:

- Rely on **APP_ENV**: set `APP_ENV=production` on the live server and `APP_ENV=local` (or similar) in dev; Stripe will use live keys in production and test keys otherwise.
- Or set **STRIPE_MODE** explicitly so Stripe doesnÔÇÖt follow APP_ENV (e.g. `APP_ENV=production` but `STRIPE_MODE=test` for testing payments on prod).

## .env variables

### Dev / test mode (`STRIPE_MODE=test` or `APP_ENV` Ôëá production)

- `STRIPE_SECRET_TEST` ÔÇô Stripe **secret** key (starts with `sk_test_...`).
- `STRIPE_PUBLISHABLE_KEY_TEST` ÔÇô Stripe **publishable** key (starts with `pk_test_...`).
- `STRIPE_WEBHOOK_SECRET_TEST` or `STRIPE_WEBHOOK_SECRET` ÔÇô Webhook signing secret for your **test** webhook endpoint.

### Production / live mode (`STRIPE_MODE=live` or `APP_ENV=production`)

- `STRIPE_SECRET` ÔÇô Stripe **secret** key (starts with `sk_live_...`).
- `STRIPE_PUBLISHABLE_KEY_LIVE` ÔÇô Stripe **publishable** key (starts with `pk_live_...`).
- `STRIPE_WEBHOOK_SECRET_LIVE` or `STRIPE_WEBHOOK_SECRET` ÔÇô Webhook signing secret for your **live** webhook endpoint.

### Optional (any mode)

- `STRIPE_SUCCESS_URL` ÔÇô Where to send the user after successful checkout (default: app URL + `/credits`).
- `STRIPE_CANCEL_URL` ÔÇô Where to send the user if they cancel checkout.
- `STRIPE_WEBHOOK_URL` ÔÇô Your Stripe webhook URL (for reference; Stripe dashboard uses this).

## Example .env snippets

**Local dev (test keys):**

```env
APP_ENV=local
STRIPE_MODE=test
STRIPE_SECRET_TEST=sk_test_xxxxx
STRIPE_PUBLISHABLE_KEY_TEST=pk_test_xxxxx
STRIPE_WEBHOOK_SECRET_TEST=whsec_xxxxx
# optional:
STRIPE_SUCCESS_URL=http://localhost:8082/credits
STRIPE_CANCEL_URL=http://localhost:8082/credits
```

**Production (live keys):**

```env
APP_ENV=production
STRIPE_MODE=live
STRIPE_SECRET=sk_live_xxxxx
STRIPE_PUBLISHABLE_KEY_LIVE=pk_live_xxxxx
STRIPE_WEBHOOK_SECRET_LIVE=whsec_xxxxx
STRIPE_SUCCESS_URL=https://yoursite.com/credits/success
STRIPE_CANCEL_URL=https://yoursite.com/credits
```

If you omit `STRIPE_MODE`, production servers should have `APP_ENV=production` and the correct live Stripe vars; local can use `APP_ENV=local` and test vars.

## Using the keys in code

- **Secret / API**: `config('services.stripe.secret')` (used for Checkout and webhooks).
- **Publishable**: `config('services.stripe.publishable')` (for frontend Stripe.js if you add it).
- **Webhook secret**: `config('services.stripe.webhook_secret')`.
- **Mode**: `config('services.stripe.mode')` Ôćĺ `'test'` or `'live'`.

All Stripe usage in the app should go through `config('services.stripe.*')`; avoid reading Stripe keys from `env()` directly.

## 8. Spreadsheet parity plan

---

## GASQ Spreadsheet Parity Plan
source_spreadsheet: GASQ_TCO_Model_TCO_Draft_Updated_v12.xlsx
goal: Pixel-perfect UI + exact spreadsheet-matching calculations
status: active

## Objective

Make the GASQ web app match `**GASQ_TCO_Model_TCO_Draft_Updated_v12.xlsx**`:

- **UI/UX**: pixel-perfect to the established calculator UI.
- **Logic**: calculator outputs match spreadsheet outputs exactly for the same inputs.

This plan is intentionally incremental: we will enable backend functionality **one route at a time**, with testable spreadsheet parity at each step.

## Current State (as of this plan)

- Laravel serves the bundled SPA for many calculator/onboarding routes to keep UI pixel-perfect.
- Duplicate route conflicts were cleaned up; backend endpoints were moved under `/_backend/*` to avoid collisions.
- Spreadsheet workbook is present at repo root: `GASQ_TCO_Model_TCO_Draft_Updated_v12.xlsx`.

## Principles

- **Spreadsheet is the source of truth**: any mismatch is treated as a bug until proven otherwise.
- **Golden-test driven**: every calculator gets a fixed set of input cases and expected outputs.
- **No silent drift**: parity checks must run locally before shipping changes.
- **One calculator at a time**: enable functionality only for one page/route per iteration.

## Scope: What ÔÇťParityÔÇŁ Means

For each calculator route:

- **Inputs**: all user-editable fields map to a known spreadsheet cell(s).
- **Outputs**: the displayed KPI/results map to specific spreadsheet output cells.
- **Rounding/formatting**: match spreadsheet rounding rules (including displayed decimals and currency formatting).
- **Edge cases**: blanks, zeros, and invalid inputs behave consistently with spreadsheet assumptions.

## Phase 0 ÔÇö Lock UI Routing (already mostly done)

- Serve the SPA shell from Laravel (bundled assets under `public/assets/*`) for the calculator routes.
- Keep Laravel backend endpoints under `/_backend/*` (auth-protected) until each route is wired.

Deliverable:

- Stable list of routes that load the embedded SPA shell.

## Phase 1 ÔÇö Spreadsheet Reverse-Engineering (foundation)

### 1.1 Identify the ÔÇťauthoritative outputsÔÇŁ

Start from the main report sheets:

- `Summary`
- `Contract_Summary`
- `Government_TCO`
- `Vendor_Pricing`

For each chosen sheet:

- Enumerate required output cells (KPI numbers the site must show).
- Record formulas and referenced cells.

### 1.2 Identify user-editable inputs

Start with:

- `Inputs`
- `Scope_of_Work`
- `Post_Positions`
- supporting modules (`Vehicle`, `Uniform_Equipment`, etc.)

Deliverable:

- A mapping document: **UI field Ôćĺ Spreadsheet cell** and **UI KPI Ôćĺ Spreadsheet cell**.

## Phase 2 ÔÇö Golden Test Harness (must-have)

Create a repeatable parity runner:

- Load spreadsheet model.
- Apply a test caseÔÇÖs input values (cell assignments).
- Recalculate outputs.
- Compare expected outputs to website/backend outputs.

Notes:

- Use a ÔÇťgolden fixturesÔÇŁ folder containing 10ÔÇô30 test cases per calculator.
- Include at least:
  - 1 minimal case
  - 1 typical case
  - 1 high-volume case
  - 1 edge case (zeros/blanks)

Deliverable:

- Command that prints a clear diff of mismatches (cell-by-cell).

## Phase 3 ÔÇö Implement Backend Calculation Engine (one calculator at a time)

For each calculator route (order below):

1. Implement backend compute endpoint under `/_backend/*`.
2. Ensure it matches the spreadsheet using the golden tests.
3. Wire the SPA or Blade UI to call that endpoint (no visual changes unless required).
4. Repeat until stable, then move to the next route.

### Recommended order

1. `**/security-billing`**
2. `**/main-menu-calculator**` (tabs: security/manpower/economic/billrate)
3. `**/contract-analysis**`
4. `**/mobile-patrol-calculator**` / `**/mobile-patrol-comparison**`
5. `**/gasq-tco-calculator**` (largest model)

Deliverable per route:

- Backend endpoint returning a JSON payload of all KPIs shown on the page.
- Parity tests passing against the spreadsheet for the routeÔÇÖs golden cases.

## Phase 4 ÔÇö Production Hardening

- Centralize rounding/formatting rules (currency, percent, hours) to avoid drift.
- Add input validation aligned to spreadsheet assumptions.
- Add logging for calculation version + inputs (for debugging mismatches).
- Add versioning for spreadsheet model updates:
  - store workbook version string (e.g., `v12`)
  - tie backend calc version to workbook version

## Phase 5 ÔÇö Spreadsheet Updates Workflow

When the spreadsheet changes:

- Add the new workbook file version.
- Run the parity harness to detect changed outputs.
- Update mapping/fixtures only if the spreadsheet truly changed expectations.

## Definition of Done

We consider parity complete when:

- All target calculator routes use backend endpoints for computations.
- Golden tests pass for every calculator route against the spreadsheet workbook.
- UI remains pixel-perfect to the reference calculator design.
