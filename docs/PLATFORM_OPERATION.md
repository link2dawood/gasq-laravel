# GASQ Platform — How the System Works

> **Consolidated copy:** the same material (and the rest of the docs) lives in **[docs/README.md](README.md)**. This file is kept for edit history and deep links.

This document describes how the GASQ Laravel application behaves today: user journeys, security gates, calculators, billing, and supporting services. Use it for onboarding, client walkthroughs, and technical handoff.

---

## 1. What this application is

GASQ Laravel is a **web application** for the GASQ security-services marketplace. It provides:

- **Marketing and informational pages** (landing, pricing, FAQ, pay scale, payment model).
- **User accounts** (buyers, vendors, admins) with profile and wallet **credits**.
- **Job board** (post jobs, bids, vendor profiles).
- **Security and labor calculators** implemented as **Blade + Bootstrap** pages with **server-side math** (PHP “V24” compute engines), not a React shell for the main calculator experience.
- **Monetization** via **Stripe** checkout to add credits to a user wallet, and **per-run deduction** of credits when a calculator **compute API** succeeds.
- **SMS OTP** for **phone verification** after registration (Twilio, custom codes stored in the database).

Core persistence uses **Laravel + Eloquent** (MySQL or other configured database). Session-based authentication is standard Laravel `Auth`.

## 2. Technology stack

| Area | Choice |
|------|--------|
| Framework | Laravel (PHP 8.2+) |
| UI | Blade templates, Bootstrap 5, shared layout (`resources/views/layouts/app.blade.php`) |
| Auth | Laravel session auth, optional Google OAuth (Socialite), password reset |

## 3. Roles

- **Buyer**: Purchases security services; can post jobs and use calculators per business rules.
- **Vendor**: Can bid and use calculators; **not** required to post a job to unlock calculators.
- **Admin**: Analytics, settings, FAQ, token/wallet management (routes under `admin` + `EnsureUserIsAdmin`).

User model helpers (e.g. `isBuyer()`, `isVendor()`, `isAdmin()`) support middleware and policies.

## 4. End-to-end user flow (buyer-centric)

This is the intended **happy path** for someone who wants to use paid calculators:

1. **Register / log in**  
   Standard Laravel registration (buyer vs vendor per registration path / `user_type`).

2. **Verify phone (SMS OTP)**  
   After signup, the user completes **`/phone/verify`**: codes are generated by the app, stored (e.g. hashed) in **`verification_codes`**, and sent via **Twilio**. Middleware **`phone.verified`** blocks downstream features until `users.phone_verified` is set.

3. **Purchase credits**  
   User opens **`/credits`**, chooses a **pricing plan**, and pays with **Stripe Checkout**. Webhook **`/stripe/webhook`** confirms payment and **`WalletService`** credits the wallet; transactions are recorded for history and receipts.

4. **Post at least one job (buyers only)**  
   Middleware **`buyer.has_job`** requires that a **`job_postings`** row exists for that user before calculator **page** routes in the gated group are allowed. Vendors skip this requirement.

5. **Use calculators**  
   User opens a calculator URL (see section 6). The page loads inputs; changing inputs triggers **POST** requests to **`/_backend/.../compute`** endpoints. On **success**, **`CalculatorRunBillingService`** deducts **`config('credits.calculator_per_run')`** credits (from **`.env`**: **`CALCULATOR_CREDITS_PER_RUN`**, default 5, minimum 1) inside a **database transaction** so a failed engine run does not consume credits.

6. **Review balance**  
   **`/account-balance`** shows transaction history; **`/credits/***` supports buying more.

**Credits page copy** explains that each calculator run uses the configured number of credits (see `resources/views/credits/index.blade.php`).

## 5. Middleware and access control

Aliases registered in **`bootstrap/app.php`**:

| Alias | Purpose |
|-------|--------|
| `phone.verified` | Redirects to phone verification if not verified. |
| `has.credits` | Requires wallet balance **≥** `config('credits.calculator_per_run')` so users cannot open calculator UIs without enough credits for one run. |
| `buyer.has_job` | For **buyers** only, requires at least one `JobPosting`; vendors pass through. |

**Important routing split** (see **`routes/web.php`**):

- **Calculator Blade pages** (main menu, contract analysis, cost analysis, bill rate, instant estimator index, etc. **inside** the group at line ~81) use:  
  `auth` + `phone.verified` + `has.credits` + `buyer.has_job`.
- **`/_backend/.../compute`** POST routes live under **`auth` + `phone.verified`** only (same session as the app). Credits are still enforced at **deduction** time; buyers who bypass the UI would not be blocked by `has.credits` or `buyer.has_job` on the HTTP route alone—**tightening** that (e.g. attaching the same middleware to compute routes) is a recommended hardening step if direct API access is a concern.

**Public calculator-related pages** (no login required for **view**): examples include **`/security-quote`** (marketing preview; no server quote math), **`/calculator`** hub, **`/gasq-tco-calculator`** **view**, **`/post-job`** informational, etc. **Compute** for those tools still requires an authenticated, verified user when using the backend API.

## 6. Calculator surface area

### 6.1 Gated full calculators (typical “app” calculators)

These **GET** routes expect the full middleware stack on the **page** (auth, phone, credits, buyer job):

- Main menu, contract analysis, security billing, mobile patrol calculator & comparison  
- Cost analysis, bill rate analysis, manpower hours, economic justification  
- Hourly pay, budget, unarmed guard services  
- Mobile patrol **analysis** & **global security pricing**  
- Mobile patrol hit calculator, buyer fit index  
- Workforce appraisal report (and aliases CFO / post position / appraisal comparison)  
- GASQ direct labor build-up, GASQ additional cost stack  

Navigation is centralized in partials such as **`resources/views/partials/nav-calculators-dropdown.blade.php`**.

### 6.2 Other entry points

- **`/gasq-instant-estimator`**: Combined **public** GET with **`/instant-estimator`** redirect; **compute** uses **`/_backend/instant-estimator/compute`** (authenticated).
- **Static/marketing** pages: **`/post-coverage-schedule`**, **`/open-bid-offer`**, **`/vendor-form`**, **`/register/buyer`**, **`/register/vendor`**, etc., are largely Blade content (some link into authenticated job creation).

### 6.3 Backend compute endpoints

Authenticated **`POST`** JSON payloads (`version: v24` where applicable, `scenario` array) hit controllers under **`App\Http\Controllers\Backend\`**, for example:

- **`StandaloneV24ComputeController`** — `type` in route must match **`StandaloneV24ComputeService`** (cost analysis, bill rate, MPA, GSP, workforce appraisal, TCO, absorbed rate, government, keeps doors open, unarmed, buyer fit, direct labor build-up, additional cost stack, etc.).
- **`InstantEstimatorComputeController`**, **`MainMenuComputeController`**, **`ContractAnalysisV24ComputeController`**, **`MobilePatrolV24ComputeController`**, **`SecurityBillingV24ComputeController`**, **`SecurityBillingComputeController`**.

Successful responses include calculator payload plus **`credits_spent`** and **`credits_remaining`** (when billing applies). **HTTP 402** JSON is returned when the wallet cannot cover the run.

## 7. Credits and configuration

- **Wallet**: `wallets.balance` per user; **`WalletService`** for **add** (purchase, bonus, admin) and **spend**.
- **Transactions**: Ledger rows for audit (`transactions` table).
- **Per-run cost**: **`config/credits.php`** ← **`CALCULATOR_CREDITS_PER_RUN`** in **`.env`** (documented in **`.env.example`**).
- **Stripe**: See **`docs/STRIPE-ENV.md`** and **`StripeCreditsController`**; success flow may nudge buyers toward **`jobs.create`** if they have credits but no job yet (see **`CreditsController`**).

Admin can inspect/adjust wallets via **`/admin/tokens`** ( **`AdminTokensController`** ).

---

## 8. Phone verification (OTP)

- Registration collects **phone**; after account exists, user completes OTP at **`/phone/verify`**.
- **Channel**: SMS only via Twilio; **provider mode**: **custom** codes generated in-app and persisted (**not** Twilio Verify API for the code itself).
- Env vars: **`TWILIO_ACCOUNT_SID`**, **`TWILIO_AUTH_TOKEN`**, **`TWILIO_FROM`** (see **`.env.example`**).

## 11. Summary for stakeholders

**GASQ Laravel** delivers a **buyer–vendor marketplace** with **credit-based access** to **professional security calculators**. Buyers **verify their phone**, **buy credits with Stripe**, **post a job**, then use **native Blade calculators** backed by **server-side PHP engines**. Each **successful** calculator **API run** costs **`CALCULATOR_CREDITS_PER_RUN`** credits (configurable). Vendors can use calculators without posting a job. The system is **documented, testable**, and **operable** without relying on a React SPA for core calculator UX.
